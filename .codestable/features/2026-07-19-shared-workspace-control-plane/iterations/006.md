# Iteration 006：真实 Claude Host 握手与 Provider 复核

## 当前理解

Iteration 005 的外部条件包含旧 Workspace owner 与 Codex provider。本轮必须重新读取实际状态；旧
pid 的存在不能从历史报告推断，且 Workspace lock 恢复后可以继续验证真实 Claude Host 链路。

## 实现方式

先用 diagnostics 与 `ps` 复核本仓库 owner，再运行 fresh Codex ephemeral 请求。Workspace 空闲后，
以当前 `dist/mcp-cli.js` 作为唯一 MCP 配置启动真实 Claude CLI，要求它调用 `cs_agent_list`，并消费
完整 stream-json 直到进程结束。

## 本轮变更

- 没有代码变更。
- 旧 Workspace owner 已恢复，不再作为当前 blocker 条件。
- 增加真实 Claude Host 对新版 stdio/Broker 握手、13 tools 枚举及退出清理证据。
- Goal 保持 `running`，恢复后的 blocked-audit count 递增为 2。

## 验证证据

- `node dist/mcp-cli.js agents list --json` 返回 diagnostics v1 且 agents 为空；`ps -p 43619` 也为空，
  证明旧 owner 已退出。
- fresh Codex ephemeral read-only 请求仍在 5 次重连后返回 401 `INVALID_API_KEY`。
- Claude CLI 2.1.186 以 `--strict-mcp-config` 启动当前 `dist/mcp-cli.js`；init event 报告
  `mcp_servers: [{name: cs-agent, status: connected}]`，并列出全部 13 个 `cs_agent_*` tools。
- Claude provider 连续重试 10 次后返回 503 `No available accounts`，模型未产生 tool call。
- Claude Host 退出后 diagnostics 仍为空，当前 `dist` 没有遗留 `--internal-broker` 进程。

## 遇到的问题

Codex provider credential 仍无效；Claude provider 暂无可用账户。真实 MCP transport 已建立，但没有
模型响应就无法证明 create/list/status/wait/cancel/destroy 的跨 Host 行为。仅凭 connected 与 tools
枚举不足以通过 C16。
